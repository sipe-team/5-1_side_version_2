## 2. 기술 방향 논의

### 배경

이전 메인테이너 의견:

> vanilla extract로 headless한 컴포넌트로 하려다가 나중에는 그냥 headful하게 가버렸는데,
> 개인적으로 요즘 대세인 Radix나 shadCN registry로 만드는 것도 좋은 것 같다.
> Radix/shadCN 밑에다 붙이고, 사이프 색깔 쉽게 묻히는 개발자 간지나는 컴포넌트를 만들자.

### 선택지 3가지

#### A. 현재 방식 유지

CSS Modules + 자체 구현을 계속 이어간다.

- **장점**
  - 이미 11개 컴포넌트가 구현되어 있어 추가 학습 비용 없음
  - 외부 의존성 최소화
- **단점**
  - Dialog, Select, Tabs 등 복잡한 컴포넌트를 직접 구현해야 함 (접근성, 키보드 인터랙션 등)
  - 동아리 리소스로 Radix 수준의 품질 달성이 어려움

#### B. Radix primitives 도입 + 스타일링은 CSS Modules 유지

복잡한 컴포넌트만 Radix primitives 위에 구축하고, 스타일링 방식은 현재를 유지한다.

- **장점**
  - 기존 코드 변경 최소화
  - 접근성/인터랙션은 Radix에 위임하면서 스타일은 자체 관리
  - Tailwind 학습 부담 없음
- **단점**
  - shadcn registry 생태계를 활용하기 어려움
  - 요즘 트렌드와는 다소 거리가 있음

#### C. Radix + Tailwind + shadcn 커스텀 registry (이전 메인테이너 제안)

shadcn 방식으로 전면 전환. `npx shadcn add @sipe-team/button` 으로 설치하는 구조.

- **장점**
  - 개발자 경험(DX)이 가장 좋음
  - 복잡한 컴포넌트도 빠르게 구축 가능
  - 트렌드에 부합, 채택률 높일 수 있음
  - 사용자가 소스코드를 직접 소유하므로 커스터마이징 자유도 높음
- **단점**
  - 기존 CSS Modules 코드를 Tailwind로 재작성해야 함
  - Tailwind를 모르는 멤버는 학습 필요
  - npm 패키지 배포 방식과 달라서 기존 CI/CD 재구성 필요

### 논의 포인트

> 1. 세 가지 선택지 중 어떤 방향이 좋은가?
> 2. Tailwind 경험이 있는 멤버가 얼마나 되는가?
> 3. 기존 11개 컴포넌트를 마이그레이션할 것인가, 새로 시작할 것인가?
> 4. 배포 방식은? (npm 패키지 vs shadcn registry vs 둘 다)

## 6. 미션 repo 운영 방식

작업 기록과 논의를 쌓아가기 위한 별도 repo를 만들자는 제안입니다.

### 구조 (안)

```
sipe-side-mission/
├── README.md              # 미션 소개, 멤버, 링크
├── members/
│   └── {이름}.md           # 개인별 주간 활동 기록 (역순)
├── weeks/
│   └── week-{N}.md         # 주차별 공통 목표 & 회의록
├── discussions/
│   └── {번호}-{주제}.md     # 주요 의사결정 기록
└── references/
    └── useful-links.md     # 참고 자료 모음
```

### 운영 규칙 (안)

- 매주 자기 `members/이름.md`에 활동 내용을 PR로 올린다
- 주요 기술 결정은 `discussions/`에 기록한다
- 회의록은 `weeks/`에 정리한다 (회의록 정리는 매주 돌아가면서 하는 것으로 정해봐요)

### 논의 포인트

> 1. 이 구조가 괜찮은가? 추가/삭제할 부분이 있는가?
> 2. PR 리뷰를 필수로 할 것인가? (몇 명이 approve 해야 머지를 할 것인가?)
> 3. Test code
> 4. sipe.team (사이프 공식 홈페이지) 에도 적용을 할 것 인가? (어차피 언젠가는 해야 함)
